# Workout Execution Flow Design

## Purpose

This document defines how a previewed `ApprovedWorkoutPlan` should eventually become a user-executable workout flow.

The goal is to preserve the current preview-first architecture while adding a clear future path for selecting a plan, logging actual work, completing the workout, and comparing planned work against actual performance.

This is a design document only. It does not implement database schema changes, endpoints, Streamlit execution controls, persistence, or workout-plan execution.

## Current stable architecture

The current workout plan preview flow is:

```text
UserHealthState
→ CoachingDecision
→ TrainingConstraints
→ WorkoutConstraints
→ EquipmentProfile
→ WorkoutContext
→ CandidateWorkoutPlan
→ workout validator
→ ApprovedWorkoutPlan
→ deterministic workout renderer
→ Streamlit Workout Plan Preview
```

The current endpoint remains stateless:

```text
GET /workout-plans/preview/{user_id}
```

A preview should not create durable records. Durable records should only be created when the user explicitly selects or starts a workout plan.

## Design principles

1. Preview is not execution.
2. Manual workout logging must continue to work independently.
3. The execution flow should reuse or extend the existing workout logging model instead of replacing it.
4. Planned work and actual work should be stored separately enough to support planned-vs-actual comparison.
5. `ApprovedWorkoutPlan` should remain the renderable plan contract.
6. Future AI/CrewAI workout generation should not be involved in this flow until deterministic execution is stable.
7. The system should allow abandoned or partially completed sessions without corrupting workout history.

## Future lifecycle

A workout plan execution should move through a clear lifecycle:

```text
previewed
→ selected
→ started
→ in_progress
→ completed
```

Optional terminal state:

```text
abandoned
```

### previewed

A preview is generated by `GET /workout-plans/preview/{user_id}`.

Characteristics:

- stateless
- not persisted
- safe to regenerate
- not treated as a commitment by the user
- may change when health state, equipment profile, workout constraints, or training constraints change

### selected

The user chooses a previewed plan and asks the app to save it for execution.

Characteristics:

- creates a durable `WorkoutPlanInstance`
- stores a snapshot of the approved plan
- freezes the plan content at selection time
- does not yet create actual set logs unless the implementation chooses to create a draft session immediately

### started

The user begins the selected plan.

Characteristics:

- creates or activates a `WorkoutExecutionSession`
- optionally creates a workout session draft using the existing workout logging tables
- planned exercises and planned sets become visible as logging targets
- actual set fields remain empty until the user logs work

### in_progress

The user logs actual sets.

Characteristics:

- actual sets are attached to planned exercises when possible
- the user can add extra sets or exercises if needed
- the plan should tolerate missing actuals for skipped exercises
- the app should preserve actual logged work even if it differs from the plan

### completed

The user finishes the workout.

Characteristics:

- workout session is finalized
- planned-vs-actual comparison metadata is computed
- completed session contributes to future workout history
- completed session can inform future `WorkoutConstraints.recent_exercises`
- completed session can inform future readiness, volume, and progression logic later

### abandoned

The user cancels or abandons the execution flow.

Characteristics:

- plan instance remains available for audit/history if needed
- actual workout logs should not be deleted if the user already logged real work
- the app may mark the plan instance abandoned while preserving any manual workout session records

## Future entities and model names

These names are proposed for future implementation. They are not implemented in this milestone.

### WorkoutPlanInstance

Represents the selected approved plan snapshot.

Suggested fields:

- id
- user_id
- status
- scenario
- confidence
- approved_workout_plan_json
- training_constraints_json
- workout_constraints_json
- source
- created_at
- selected_at
- started_at
- completed_at
- abandoned_at

Suggested statuses:

- selected
- started
- in_progress
- completed
- abandoned

### PlannedWorkoutExercise

Represents each planned exercise in the selected plan.

Suggested fields:

- id
- workout_plan_instance_id
- exercise_name
- movement_pattern
- equipment
- planned_sets
- planned_reps
- planned_rir_min
- planned_rir_max
- notes
- sort_order

### PlannedWorkoutSet

Optional future model if planned sets need set-level detail instead of exercise-level ranges.

Suggested fields:

- id
- planned_workout_exercise_id
- set_number
- target_reps_min
- target_reps_max
- target_rir_min
- target_rir_max
- target_load
- notes

For v1 implementation, exercise-level planned sets may be enough. Set-level planned rows can wait until the execution UI needs finer control.

### WorkoutExecutionSession

Represents the execution attempt for a selected plan.

Suggested fields:

- id
- workout_plan_instance_id
- user_id
- workout_session_id
- status
- started_at
- completed_at
- abandoned_at
- notes

This can bridge the plan instance to existing workout logging records.

### WorkoutSetActual

Represents actual performed work. This may be a new model or a mapping layer over existing workout set rows.

Suggested fields:

- id
- workout_execution_session_id
- planned_workout_exercise_id
- workout_set_id
- exercise_name
- set_number
- actual_reps
- actual_weight
- actual_rir
- completed
- notes

If existing workout set tables already capture enough detail, this can be implemented as a mapping table rather than a replacement set table.

## Persistence options

### Option A: snapshot-first persistence

Persist the full `ApprovedWorkoutPlan` as JSON on `WorkoutPlanInstance`.

Pros:

- simple
- preserves exactly what the user selected
- protects against later changes to rendering/generation logic
- supports audit and planned-vs-actual comparison

Cons:

- JSON queries are less convenient
- planned exercise details need extraction for detailed analytics

Recommended for v1.

### Option B: fully normalized planned workout tables

Persist plan, exercises, and planned sets as relational rows.

Pros:

- easier analytics
- easier planned-vs-actual joins
- more structured long-term

Cons:

- more implementation work
- harder to evolve the plan contract early

Recommended later, after the plan execution UX is clearer.

### Recommended initial persistence path

Use a hybrid:

1. Store `approved_workout_plan_json` on `WorkoutPlanInstance`.
2. Also store planned exercise rows in `PlannedWorkoutExercise`.
3. Delay `PlannedWorkoutSet` until set-level planned targets are necessary.

This preserves the approved plan exactly while still supporting execution UI and basic planned-vs-actual comparison.

## Relationship to existing workout logging

The execution flow should reuse and extend existing workout logging.

Existing manual workout logging should remain valid when:

- the user does not select a plan
- the user logs an unplanned workout
- the user adds extra exercises not in the plan
- the user ignores the preview feature entirely

The execution flow should eventually create or connect to the existing workout session representation so completed executed workouts appear in the same history as manually logged workouts.

Recommended relationship:

```text
WorkoutPlanInstance
→ WorkoutExecutionSession
→ existing workout session
→ existing workout sets
```

The existing workout logging system remains the source for actual performed workout history.

Planned data should not replace actual workout logs. Planned data should provide targets and comparison context.

## API design

These endpoint shapes are proposed for future implementation. They should not be added until the design is approved.

### Select or start a plan

Option 1: separate select and start.

```text
POST /workout-plans/{user_id}/select
POST /workout-plans/{plan_instance_id}/start
```

Option 2: combined start.

```text
POST /workout-plans/{user_id}/start
```

Recommended v1:

```text
POST /workout-plans/{user_id}/start
```

The request body should include an `approved_workout_plan` snapshot or a server-generated preview token if preview tokens are added later.

Because the preview endpoint is currently stateless, a v1 start endpoint can rebuild the preview server-side and persist that snapshot at start time.

### Get active workout plan

```text
GET /workout-plans/active/{user_id}
```

Returns the current selected, started, or in-progress plan instance if one exists.

### Update actual set data

Possible endpoint:

```text
PATCH /workout-plans/executions/{execution_session_id}/sets/{actual_set_id}
```

Alternative endpoint:

```text
POST /workout-plans/executions/{execution_session_id}/sets
```

Use PATCH for editing existing actual set rows and POST for adding extra actual sets.

### Complete workout

```text
POST /workout-plans/executions/{execution_session_id}/complete
```

Finalizes the workout execution session and related workout session.

### Abandon workout

```text
POST /workout-plans/executions/{execution_session_id}/abandon
```

Marks the plan execution abandoned without deleting actual workout logs.

### Planned-vs-actual summary

```text
GET /workout-plans/executions/{execution_session_id}/summary
```

Returns comparison metadata such as:

- planned exercises completed
- planned exercises skipped
- extra exercises added
- planned sets vs actual sets
- planned RIR vs actual RIR
- notes on major deviations

## How the preview endpoint fits

The existing preview endpoint remains:

```text
GET /workout-plans/preview/{user_id}
```

It should remain:

- stateless
- deterministic
- safe to regenerate
- not durable
- not a workout session draft

The preview response can be used by the UI to show the plan and let the user decide whether to start it.

A future start endpoint may either:

1. accept the approved workout plan snapshot from the client and validate it, or
2. rebuild the approved plan on the server and persist the server-generated version.

Recommended v1: rebuild on the server when starting, because it avoids trusting client-provided plan JSON.

## Planned-vs-actual comparison

A future comparison service should compute:

- exercise completion
- planned vs actual set count
- planned vs actual reps
- planned RIR range vs actual RIR
- skipped planned exercises
- added unplanned exercises
- session completion status
- notes or flags for future coaching

Examples:

```text
Planned: Goblet Squat, 2 sets x 8-10 reps, RIR 2-4
Actual: 2 sets x 10 reps, RIR 3
Result: completed within plan
```

```text
Planned: Cable Row
Actual: skipped
Result: skipped planned exercise
```

```text
Planned: none
Actual: Added Dumbbell Curl
Result: extra unplanned exercise
```

The comparison result can later feed:

- adherence scoring
- progression recommendations
- exercise selection
- recent exercise avoidance
- fatigue/readiness interpretation

## Integration with workout_service.py

`workout_service.py` should continue to own actual workout logging behavior.

Future execution-flow code should call existing workout logging functions or a small extension layer rather than duplicating session/set insertion logic.

Recommended future service boundaries:

```text
workout_plan_service.py
- builds previews
- validates candidate workout plans
- approves workout plans
- renders previews

workout_execution_service.py
- starts approved plan instances
- creates execution sessions
- links planned exercises to actual logs
- computes planned-vs-actual summaries

workout_service.py
- persists actual workout sessions
- persists actual workout sets
- retrieves workout history
```

If a new `workout_execution_service.py` is added later, it should orchestrate between persisted plan instances and the existing workout logging service.

## Preserving manual logging

Manual logging should remain a first-class workflow.

A user should be able to:

- log a workout without selecting a plan
- edit or delete manually logged data using existing flows
- generate future previews based on manually logged workout history
- use workout plan execution only when desired

The system should not require every workout to originate from an `ApprovedWorkoutPlan`.

## Staged implementation plan

### Stage 1: Design

Add this document only.

### Stage 2: WorkoutPlan persistence schema v1

Add tables for:

- workout_plan_instances
- planned_workout_exercises
- workout_execution_sessions

Optional later:

- planned_workout_sets
- workout_set_actual mappings

### Stage 3: Select/start plan endpoint

Add a start endpoint that:

1. builds the current approved preview server-side
2. persists the approved plan snapshot
3. creates planned exercise rows
4. creates a workout execution session
5. returns the active plan

### Stage 4: Convert to workout session draft

Create or link a workout session draft using the existing workout logging model.

The draft should not count as completed workout history until the user completes it.

### Stage 5: Log actual sets against planned exercises

Add set logging endpoints or UI controls that let the user record:

- reps
- weight/load
- RIR
- completion/skipped status
- notes

### Stage 6: Complete workout

Finalize execution and persist actual workout history.

### Stage 7: Planned-vs-actual summary

Add summary endpoint and future UI.

## Test strategy

Future tests should cover:

### Preview remains stateless

- calling preview does not create workout plan instance rows
- repeated preview calls are safe

### Start creates durable plan instance

- start creates `WorkoutPlanInstance`
- start stores approved plan snapshot
- start creates planned exercise rows
- start does not break manual workout logging

### Actual logging works

- actual sets can be logged against planned exercises
- skipped exercises can be represented
- extra unplanned exercises can be added
- manual logging still works independently

### Complete behavior

- complete transitions status to completed
- completed workout contributes to workout history
- completed execution computes planned-vs-actual summary

### Abandon behavior

- abandon transitions status to abandoned
- actual logged sets are preserved if they represent real work
- abandoned plan does not falsely count as completed

### Regression tests

- seeded users 101-105 still receive valid preview plans
- equipment profile constraints still affect previews
- daily recommendation endpoint remains stable
- full report safety tests remain stable

## Non-goals

- database schema
- API endpoints
- Streamlit execution buttons
- start/complete workout flow
- workout plan persistence
- workout plan scheduling
- weekly periodization
- automatic progression engine
- CrewAI workout generation
- meal-plan engine
